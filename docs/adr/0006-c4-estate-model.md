# ADR-0006: A generated C4 model of the lucOS estate

**Date:** 2026-06-14
**Status:** Proposed
**Issue:** [#422](https://github.com/lucas42/lucos_repos/issues/422)
**Related:** [ADR-0001](0001-greenfield-rewrite-as-convention-auditor.md) (establishes `lucos_repos` as the estate's static-config reader), [ADR-0004](0004-auto-close-audit-finding-issues.md) (audit-finding issue lifecycle — the divergence-output mechanism)

## Context

The estate has grown to **41 systems** (configy `systems.yaml`). The dependency structure *between* those systems is not written down anywhere as a whole — it lives, scattered, across five sources that each capture a different slice. Onboarding, change-impact analysis, and incident scoping all need a legible map of "what talks to what", and there is none.

The obvious answer — draw a C4 model — fails the durability test: a hand-drawn diagram rots within weeks of the next architecture change, and nobody trusts a diagram they can't see being kept current. So the requirement is not "a diagram" but **an evergreen model derived from tangible data sources**, regenerated mechanically.

### The key insight: no single source *is* the dependency graph

Each data source the estate already emits captures a *different kind of edge*. The model is their **union, typed by source** — and, valuably, the **divergence between sources is itself an audit finding**:

| C4 element | Source (verified 2026-06-14) | Nature |
|---|---|---|
| **Node set** (systems) | configy `systems.yaml` / `configy.l42.eu/systems` | static, **canonical** |
| **Deployment placement** | configy `hosts.yaml` + each system's `hosts:` | static, canonical |
| **Container decomposition** (C4 L2) | each repo's `docker-compose.yml` | static, per-repo |
| **Sync dependency edges** | `/_info` → `checks[].dependsOn` | live, **self-declared** |
| **Async event edges** | loganne `webhooks-config.json` | static config |
| **Trust/auth edges** (+ scope) | creds `linkedCredentials` | live, directed, scoped |

The first three need no live probing at all; the node set in particular is canonical and never depends on any service being up. The sync and trust layers are self-declared, so they *under-report* — but the gaps surface as findings rather than hiding.

### Why `lucos_repos` is the home

lucas42 chose `lucos_repos` (2026-06-14) over a new always-on service, folding into monitoring, or a standalone scheduled job. It is the right fit: `lucos_repos` already clones every repo and reads configy estate-wide for convention auditing (ADR-0001), so the static sources (configy, docker-compose, loganne config) are already in hand. And the divergence-as-audit output maps directly onto its existing audit-finding issue machinery (ADR-0004) — the same engine that raises a tracked issue for a convention breach can raise one for a model divergence.

The one capability `lucos_repos` lacks is **runtime probing**: it is a static reader today, and two edge layers (`/_info dependsOn`, creds `linkedCredentials`) are live. `/_info` is public and needs only an HTTP GET; creds needs a read credential. This shapes the phasing below.

## Decision

`lucos_repos` generates a C4 model of the estate as a **committed, version-controlled artifact**, regenerated by its existing sweep.

### 1. Two output formats, one model

- **Structurizr DSL** (`docs/c4/model.dsl`) is the **model of record** — the native C4-as-code format, one model projecting multiple views (System Landscape, per-subsystem slices, Deployment). It lays out a 41-node multi-edge graph far better than the alternatives and is the durable representation.
- **Mermaid** (`docs/c4/landscape.md`) is a **GitHub-native view** of the connected core, viewable inline with no build step. Mermaid auto-layout on the *full* estate is a hairball, so Mermaid is reserved for sub-views; the DSL is the source of truth.

### 2. Source → edge-type mapping is the model contract

The mapping in the Context table above *is* the contract. Each edge in the model is **typed by the source it came from** (sync / async / trust / deployment), and rendered distinctly.

### 3. Modelling rules

- **The node set is canonical from configy.** A system unreachable at probe time keeps its node (its declarations marked unavailable); it is **never dropped**. Liveness affects edges, never the node set.
- **loganne's own `/_info dependsOn` on its consumers is the broker / alert-suppression relationship, not a sync dependency.** loganne declares `dependsOn` on its six consumers so monitoring suppresses loganne's alert while a consumer deploys — it does *not* synchronously call them. Model this as the async/broker layer, **not** a sync edge, to avoid double-counting loganne's fan-out.
- **Divergence between sources is a finding**, routed through the existing audit-finding issue mechanism (ADR-0004) — e.g. a `linkedCredential` with no matching `dependsOn`, a loganne consumer absent from configy, or a `/_info` `system` value that disagrees with its configy key.

### 4. Phasing

- **Phase 1 (no new privilege):** ship from the five static/public sources — configy, docker-compose, loganne config, and a public `/_info` probe. This is the whole model except trust edges.
- **Phase 2:** add the **trust-edge layer** from creds `linkedCredentials`. This needs a **scoped creds read credential** for `lucos_repos` — a new privilege, hence deferred to its own step.

### 5. The artifact is the changelog

The model is committed under `docs/c4/` in `lucos_repos`. The **git history of that artifact becomes the estate's architecture changelog** — you can see the exact commit where an edge appeared or vanished, which is precisely the property a hand-drawn diagram can never give.

A working **prototype generator** (`docs/c4/prototype-generator.py`) and the **first-cut artifact** it produced (model, landscape, divergences) are committed alongside this ADR, so the approach is concrete and reproducible and the rendered diagram is viewable today. The prototype is the executable specification for the Go port into the sweep (follow-up 1); it is not the production implementation.

## Consequences

### Positive

- **A legible, always-derived estate map**, with git history as the architecture changelog.
- **No new always-on service** to deploy, monitor, back up, or firewall. The cost of a stale model is a slightly-old doc, which does not justify a 24/7 system.
- **Reuses endpoints designed for estate-wide consumption** (configy `/systems`, `/_info`) and sources `lucos_repos` already reads.
- **Divergence-as-audit:** a documentation artifact that doubles as a drift detector. The first run already surfaced three system-name divergences (see follow-up 5).
- **Honest by construction:** self-declared sources under-report, but the gaps become findings instead of silent omissions.

### Negative / limitations

- **The model is only as complete as the declarations.** `/_info dependsOn` and `linkedCredentials` are self-declared, so sync and trust layers under-report. Mitigated — gaps surface as audit findings (ADR-0004), not silent blanks.
- **The async *producer* side is not captured by any current source.** loganne config gives event *consumers*, never who *emits* each event. Producer edges need either a new convention (systems declaring emitted events) or static analysis — deferred (follow-up 4); until then the async layer is consumer-only, and this is a documented limitation, not a hidden gap.
- **Layout at estate scale.** A 41-node, multi-edge-type Mermaid graph is unreadable; this is *why* Structurizr is the model of record and Mermaid is confined to sub-views.
- **`lucos_repos` gains a runtime-probe capability** — it was a pure static reader. The new surface is modest (an HTTP GET of public `/_info` endpoints), but it is new.
- **Trust edges require a creds read credential** — a new privilege for `lucos_repos`, deferred to phase 2.
- **Committed generated output goes stale between sweeps** until the generator is wired into the CI sweep (follow-up 1). The seed artifact is dated and labelled accordingly.

## Follow-up actions

Each becomes a tracked GitHub issue on ADR sign-off (cross-referencing this ADR and [#422](https://github.com/lucas42/lucos_repos/issues/422)):

1. **Implement C4 generation in the `lucos_repos` Go sweep** — port `prototype-generator.py`: read the static sources already in hand + add a public `/_info` probe step; emit the Structurizr DSL + Mermaid; commit the artifact. (Phase 1.)
2. **Route source divergences through the audit-finding issue mechanism** (ADR-0004), so model drift raises a tracked issue exactly as a convention breach does.
3. **Phase 2 — trust-edge layer:** provision a scoped creds *read* credential for `lucos_repos`, then add `linkedCredentials` edges to the model.
4. **Async producer-side edges:** decide a convention for systems to declare the events they emit (or adopt static analysis); until then the async layer is documented as consumer-only.
5. **Triage the three system-name divergences** found on the first run — `lucos_mail` reports `lucos_mail_docs`, `tfluke` reports `tfluke_app`, `lukeblaney_co_uk` reports `lukeblaney.co.uk` — as intentional or as drift to correct.
