# ADR-0001: Greenfield Rewrite as Convention Auditor

**Date:** 2026-03-05
**Status:** Accepted
**Issue:** [#22](https://github.com/lucas42/lucos_repos/issues/22)

## Context

lucos_repos had become a shell service: a Node.js server with a deprecated webhook endpoint and a `/_info` response. Its original purpose (calculating version numbers for deploys) was superseded when that logic moved into the `lucos/deploy` CircleCI orb. The service had no meaningful function, but the subdomain and deployment slot remained.

lucas42 wanted to reimagine lucos_repos from the ground up as a tool that audits all lucas42 GitHub repositories against a set of well-defined conventions, raises issues on non-compliant repos, and provides a dashboard showing compliance state.

Key constraints from the outset:
- **Deterministic only.** No LLM involvement in auditing -- checks must be repeatable and predictable.
- **No auto-fix.** The service raises issues; agents or humans fix them. The audit verifies on the next sweep.
- **Clear attribution.** Issues raised by the service must be visibly machine-generated, not appear to come from a personal account.

## Decision

### Tech stack: Go + SQLite

The service will be rewritten in Go with SQLite for storage.

**Go** was chosen because:
- Proven in the lucos ecosystem (lucos_media_metadata_api, lucos_creds)
- Excellent standard library for HTTP servers and JSON handling
- Mature GitHub API client ecosystem (`google/go-github`)
- Single static binary produces small, fast Docker images
- No runtime dependency management (unlike Node.js/npm or Python/pip)
- A boring, proven choice -- appropriate for an audit tool

**SQLite** was chosen because:
- No additional container needed (Postgres would be overkill for ~40 repos and ~10 conventions)
- Proven in the ecosystem (lucos_media_metadata_api, lucos_creds)
- Easily backed up as a single file
- The volume is registered in lucos_configy with `recreate_effort: automatic` since all data can be regenerated by re-running audits

Node.js (the current stack) was considered but rejected. The existing codebase is 66 lines with no meaningful logic to preserve, and Go is a better fit for a service whose primary job is structured API calls and data storage.

### Single container

One Go binary serving HTTP. No worker process, no message queue. The audit workload is small enough (~40 repos checked against ~10 conventions) that it runs synchronously within a scheduled task inside the same process.

### Authentication: GitHub App

A dedicated GitHub App (not a Personal Access Token) authenticates the service for GitHub API access. This was a modification from the architect's original proposal, which suggested starting with a PAT for simplicity.

lucas42's reasoning: issues raised by the audit service must be clearly attributable to an automated tool, not appear to come from a personal account. A GitHub App provides a distinct bot identity, consistent with how the agent personas authenticate elsewhere in the lucos ecosystem.

Required permissions:
- `Contents: Read` (check file existence and content)
- `Issues: Read & write` (raise and track compliance issues)
- `Administration: Read` (check branch protection rules, repo settings)
- `Metadata: Read` (standard)

The Go binary embeds its own JWT signing and installation token exchange logic (~50 lines), rather than depending on the `get-token` script used in the coding sandbox.

### Conventions defined in code

Each convention is a Go function, not a config file or DSL entry. Conventions are registered in code and return a structured result:

```go
type ConventionResult struct {
    Convention string
    Pass       bool
    Detail     string
}

type Convention struct {
    ID          string
    Description string
    Check       func(repo RepoContext) ConventionResult
}
```

This was chosen because:
- Convention checks are procedural and benefit from the expressiveness of a real programming language
- Adding a convention means writing a function and registering it -- no indirection through config parsing
- Each convention can declare its own applicability (e.g. Docker conventions only apply to repos with a Dockerfile)

### Scheduled auditing, not event-driven

A full audit sweep runs on a timer (every 6 hours), implemented as a goroutine inside the Go process. On startup, an immediate sweep runs, then repeats on the interval.

Webhooks were rejected because:
- Setting up webhooks on every repo is itself a convention to enforce (chicken-and-egg problem)
- Webhooks are unreliable (missed deliveries, race conditions)
- The audit is cheap (~40 API calls per convention per sweep)
- A schedule is simpler, more predictable, and inherently self-healing

### Repo discovery: GitHub API + lucos_configy

The service queries the GitHub API for all repos in the lucas42 account, then cross-references with data from lucos_configy. This was a modification from the architect's original proposal, which only used the GitHub API.

lucas42's addition: lucos_configy data enables two things:
1. A convention that checks repos appear in configy (and vice versa)
2. Using configy metadata (e.g. system vs component classification) to determine which conventions apply to which repos

### Issue management

When a convention fails for a repo:
1. Check if an open issue already exists for that convention (identified by a convention tag in the title or body)
2. If no open issue exists, create one with a standardised title format
3. Store the issue URL in the findings table

When a convention subsequently passes, the service does **not** auto-close the issue. The interaction between audit-raised issues and GitHub's auto-close behaviour (from PR closing keywords) requires separate design work, tracked in [#30](https://github.com/lucas42/lucos_repos/issues/30).

### Dashboard and API

The same Go binary serves:
- `GET /` -- Server-rendered HTML compliance matrix (repos vs conventions, colour-coded, with issue links). Plain `html/template`, no JavaScript framework.
- `GET /api/status` -- JSON version of the compliance data for programmatic consumption.
- `GET /_info` -- Standard lucos info endpoint with checks (e.g. "last audit completed successfully") and metrics (e.g. "total violations", "compliant repos").

## Consequences

### Positive

- **Clear purpose.** The service goes from being a shell to having a well-defined, useful role in the ecosystem.
- **Low operational overhead.** Single container, SQLite storage, no external dependencies beyond the GitHub API and lucos_configy.
- **Incrementally deliverable.** The implementation is broken into 8 tickets (#23-#30), with a working read-only audit tool achievable after the first 4.
- **Self-healing.** Scheduled sweeps mean missed events or transient failures are automatically corrected on the next run.
- **Extensible.** New conventions are added by writing a Go function -- no framework changes needed.

### Negative

- **Complete rewrite.** The existing Node.js codebase is discarded entirely. There is no migration path -- this is a clean-slate replacement. (Mitigated by the fact that the existing codebase has no meaningful logic.)
- **6-hour detection lag.** Convention violations are not detected until the next scheduled sweep. This is acceptable for the current use case but would need revisiting if near-real-time detection became important.
- **GitHub App complexity.** Embedding JWT signing in the Go binary is more complex than using a PAT. The trade-off is justified by clearer attribution and consistency with the rest of the ecosystem.
- **Auto-close interaction is unresolved.** The relationship between audit-raised issues and GitHub's auto-close behaviour needs separate design work (#30). Until that is resolved, there is a risk of issues being prematurely closed by PR merges.
